## 摘要

我在 30 条 HotPotQA 多跳问答样本上复现了 CoT、ReAct 与 Reflexion 三种推理范式。使用严格 Exact Match（EM）评估时，Reflexion 的准确率为 **60.0%**，显著高于 ReAct 的 **36.7%** 和 CoT 的 **43.3%**；但平均 Token 消耗也达到 **7109**，约为 CoT 的 5.5 倍、ReAct 的 1.6 倍。本文基于这次实验的原始数据，讨论 Reflexion 为什么有效、成本结构如何、在哪些答案类型上提升最明显，以及为什么它远不是“万能药”。

## 引言：当 Agent 开始“回头看”

大语言模型（LLM）像一位反应极快但记性一般的实习生：你给它一个问题，它能滔滔不绝地给出答案，却常常在一些细节处翻车。CoT（Chain-of-Thought）让这位实习生“边想边说”，ReAct 进一步让它可以查阅资料（Wikipedia），但两者都有一个共同缺陷——**一旦开始犯错，就会沿着错误一路狂奔**。

Reflexion 的核心想法很简单：让 Agent 在失败后“回头看一眼”，用语言总结一下刚才哪里做错了，然后在下一次尝试时带上这份“反思”。这听起来像人类的学习方式，但在 LLM 上，它到底是真的学到了，还是只是换了一种更精致的提示工程？本文通过一次小规模但完整的复现，尝试给出基于数据的回答。

## 实验设置

- **数据集**：HotPotQA `fullwiki` 验证集，按 `seed=42` 随机打乱后取前 30 条。
- **模型**：Kimi Code（`kimi-k2p5-coding`），通过 OpenAI 兼容接口调用。
- **工具**：Wikipedia API（MediaWiki）用于 ReAct 和 Reflexion 的检索。
- **评估**：采用 HotPotQA 官方风格的 **Exact Match（EM）** 与 **F1**；归一化包括小写、去冠词和去标点。
- **三种范式**：
  - **CoT**：直接要求逐步推理并给出答案。
  - **ReAct**：Thought → Action（Search/Finish）→ Observation 循环，最多 8 步。
  - **Reflexion**：第一轮 ReAct；若失败，将完整轨迹与标准答案传给模型生成反思；第二轮 ReAct 携带反思重试。

完整 Notebook 与代码已开源在 [GitHub](https://github.com/ENTINSER/my-tech-blog) 的复现实验目录中。

## 核心发现：Reflexion 有效，但不是魔法

### 1. 为什么 Reflexion 比 ReAct 好？

实验结果如下：

| 方法 | EM (%) | F1 (%) | 平均 Token |
|---|---:|---:|---:|
| CoT | 43.33 | 46.95 | 1300 |
| ReAct | 36.67 | 40.78 | 4501 |
| Reflexion | **60.00** | **60.00** | **7109** |

Reflexion 的 EM 比 ReAct 高出 **23.3 个百分点**。这个数据说明：给 Agent 一次“事后复盘”的机会，确实能让它在第二轮避开第一轮犯过的错误。

但为什么第一轮 ReAct 只有 36.7%？一部分原因是 HotPotQA 的多跳问题本身很难，另一部分原因是 ReAct 的“边想边搜”机制对模型的规划和工具使用能力要求很高。Agent 常常会：
- 搜到正确页面，却没提取出关键信息；
- 在不该停的时候停了，或者过度搜索；
- 最终答案格式不符合标准（例如多加一个年份）。

Reflexion 的反思 Prompt 把“标准答案”作为监督信号，让模型对比自己的轨迹，从而指出这些具体错误。它不是让模型凭空变聪明，而是**把“知道正确答案后应该怎么看自己的错误”这一能力注入了下一轮推理**。

### 2. 成本结构：反思让第二轮更高效

Reflexion 的平均 Token 消耗是 ReAct 的 1.6 倍，看起来不便宜。但如果我们拆开朗轮 Token，会发现一个有趣的现象：

| 阶段 | 平均 Token |
|---|---:|
| 第一轮 ReAct | 3962 |
| 反思生成 | 886 |
| 第二轮 ReAct | 2261 |
| **合计** | **7109** |

第二轮 ReAct 的 Token 消耗（2261）明显低于第一轮（3962）。这说明：当 Agent 带上反思后，它不再漫无目的地搜索，而是更聚焦、更高效。反思虽然本身消耗了约 886 Token，但它“节省”了第二轮中大量冗余的搜索和推理。

这就像一个团队复盘会：开会本身花时间，但如果能让下一次执行少走弯路，总体效率反而更高。当然，这一切的前提是**反思必须真的指出了关键错误**；如果反思是泛泛而谈，第二轮照样会浪费 Token。

### 3. Pareto 前沿：Reflexion 处于非支配地位

![Reflexion Results](reflexion_results.png)

在“准确率-Token”坐标系中，Reflexion 位于 Pareto 前沿上：没有任何一种方法能在 Token 更少的情况下达到比它更高的准确率，也没有任何方法能在准确率更高时 Token 更少（在我们的三种方法范围内）。

这意味着：如果你的目标是“在有限预算内获得尽可能高的准确率”，Reflexion 是一个合理的选择；但如果你的目标是“最低成本地接受一定错误率”，CoT 反而更划算。Pareto 前沿的价值就在这里——它把“效果”和“成本”之间的权衡可视化，让工程师可以根据业务场景做决策。

## 分答案类型的深入分析

不同答案类型的表现差异很大：

| answer_type | count | CoT_EM | ReAct_EM | Reflexion_EM |
|---|---:|---:|---:|---:|
| date | 1 | 100.0% | 100.0% | 100.0% |
| location | 1 | 100.0% | 100.0% | 100.0% |
| number | 3 | 66.7% | 66.7% | 66.7% |
| other | 10 | 10.0% | 20.0% | 50.0% |
| person | 15 | 53.3% | 33.3% | 60.0% |

**Reflexion 提升最显著的是 `person` 和 `other` 类型**：
- `person` 从 33.3%（ReAct）提升到 60.0%；
- `other` 从 20.0% 提升到 50.0%。

为什么这两类提升大？我的推测是：
- **person**：人名类问题常常涉及“多个同名人物”或“人物关系”。反思能帮助 Agent 意识到“我搜对了名字但选错了人”，从而在第二轮更精确地定位。
- **other**：这一类别通常是“电影名、事件名、作品名”等非结构化实体。第一轮 ReAct 容易在格式上犯错（如加年份、加多余解释），反思能明确告诉它“标准答案只需要核心名称”。

**在 `number` 类型上没有提升**（三类都是 66.7%）。这可能是因为数字答案的判定非常严格——即使你推理过程正确，只要最后数字写错一个，EM 就是 0。反思在数字计算类错误上的修正能力有限，除非反思能明确指出“你算错了这一步”。

**需要注意的局限性**：`date` 和 `location` 各自只有 1 条样本，完全不具备统计意义。扩展到更大样本时，`person` 和 `other` 的优势可能会保持，但 `number` 很可能仍然是短板。

## 机制有效性拆解：反思到底改进了什么？

### 1. 三个成功案例归纳的典型错误模式

**案例 1：过度具体化**
- 问题：What animated movie, starring Danny DeVito, featured music written and produced by Kool Kojak?
- 标准答案：The Lorax
- 第一轮：The Lorax (2012)
- 反思：正确识别了电影，但多加了年份并发了空的 Finish 动作。
- 第二轮：The Lorax

这类错误非常典型：Agent 已经掌握了核心答案，却因为格式或冗余信息被判错。反思相当于告诉它“**停在正确答案即可，不要画蛇添足**”。

**案例 2：格式错误 / 过度解释**
- 问题：Has Stefan Edberg won more events than Édouard Roger-Vasselin?
- 标准答案：yes
- 第一轮：Yes, Stefan Edberg has won more events ... (59 total), while Roger-Vasselin ... (28 total).
- 反思：进行了冗余搜索，给出了冗长解释，而标准答案只需要 `yes`。
- 第二轮：yes

这里反思解决的是“**输出形式与任务要求不匹配**”的问题。很多 QA 任务的评估器只看最终答案，反思能帮助 Agent 对齐输出格式。

**案例 3：搜索策略局限**
- 问题：Jason Regler ... during a song built around which instrument?
- 标准答案：an organ
- 第一轮：（空）
- 反思：过度依赖 Wikipedia，没有尝试更广泛的搜索；下一次应确保最终动作格式正确。
- 第二轮：organ

这个案例说明反思有时能促使 Agent **更换策略**（虽然在本实现中工具只有 Wikipedia，但反思文本中确实出现了“应尝试 broader web search”的自我批评）。

### 2. 反思是“更好的提示工程”还是“真正的学习”？

我认为当前实现更接近前者：**反思是一种针对具体错误的动态提示增强**。

真正的学习应该具备两个特征：
1. **错误到修正的泛化**：看到 A 类错误后，能在未来遇到相似的 B 类错误时自动避免。
2. **不依赖标准答案**：人类复盘时往往不知道标准答案，而是从环境反馈（如任务失败、用户否定）中总结原因。

而 Reflexion 的反思生成依赖“标准答案 + 轨迹”，它更像是一位老师在批改作业时写下批注，学生带着批注重新做同一道题。如果下道题变了，批注的价值会大打折扣。

但即便如此，这种“针对同一问题的二次尝试”在交互式 Agent 场景中仍然很有价值：用户的一次否定反馈，可以替代这里的“标准答案”，让 Agent 在下一轮对话中修正。

### 3. 未来方向

当前反思的监督信号太强（需要标准答案）。更理想的版本应该：
- 从**环境反馈**（如任务失败、单元测试不通过）生成反思；
- 将反思**持久化到记忆库**中，跨样本复用；
- 让反思**结构化**（例如“格式错误”“搜索词错误”“过早停止”），便于检索和匹配。

这些方向正是 MemGPT、Self-Refine 等后续工作的探索重点。

## 与原始论文及更广泛文献的对话

### 1. 与 Reflexion 原论文的对比

Reflexion 原论文的主要实验场景是 **AlfWorld**（具身决策）和 **WebShop**（网页购物），HotPotQA 并不是其核心实验集。因此，将本次 60% EM 与原论文直接对比需要谨慎。

可能造成差异的因素包括：
- **模型能力**：原论文使用 GPT-4 系列，本次使用 `kimi-k2p5-coding`，两者在规划、指令遵循和工具使用上存在差距。
- **样本量**：本次仅 30 条，95% 置信区间约为 [41%, 77%]，波动较大。
- **数据集版本**：HotPotQA `fullwiki` 的检索难度和答案分布可能与原论文使用的子集不同。
- **评估方式**：原论文可能使用更宽松的匹配方式，而本次使用严格 EM。

### 2. 与 MemGPT、Self-Refine 的概念比较

- **Self-Refine**：同样依赖“生成 → 评估 → 重写”的循环，但通常用于单轮文本生成（如代码、摘要），不强调跨 episode 的记忆积累。
- **MemGPT**：通过操作系统式的虚拟上下文管理，把重要信息持久化到长期记忆中。它解决的是“上下文长度不够”的问题，而不是“从错误中学习”的问题。
- **Reflexion**：独特之处在于它把**语言化的反思**作为强化学习的替代信号，不需要外部奖励模型，也不需要可微训练。它的成本在于多轮 LLM 调用，但实现门槛很低。

如果把 Agent 的演进比作人类学习：CoT 是“边说边想”，ReAct 是“边查边做”，Reflexion 是“做错后记笔记”，MemGPT 是“有个笔记本随身带”。

## 实验局限性与诚实讨论

### 1. 30 条样本的统计不确定性

在 30 条样本上，60% EM 的 95% Wilson 置信区间大约为 **[41%, 77%]**。这意味着 Reflexion 的真实性能可能远高于 60%，也可能接近 50%。任何基于这 30 条样本的强结论都需要谨慎。

### 2. Wikipedia API 限制的潜在偏差

本次实验中，ReAct 和 Reflexion 分别只有 1 条和 2 条样本受到 Wikipedia 429 影响，比例很低，基本可以忽略。但需要注意的是：如果扩大样本量或提高调用频率，429 和 Read Timeout 可能会显著影响结果。

### 3. 严格 EM 可能低估模型能力

EM 要求归一化后完全相等，这对近义答案非常苛刻。例如：
- 标准答案：an organ
- 模型输出：organ

本次 F1 为 60%，EM 也是 60%，说明模型在答对时通常能完整命中，但在一些边界情况（如冠词、复数、大小写）上仍有损失。F1 作为辅助指标，能部分缓解 EM 的严苛性。

## 对工程师的实用建议

### 1. 什么时候该用 Reflexion？

基于本次实验，我建议在满足以下条件时考虑 Reflexion：
- 任务允许**多轮交互**或**重试**（例如代码生成、复杂 QA、自动化流程）。
- 错误可以被**明确评估**（有标准答案、单元测试或人工反馈）。
- 单次调用的准确率不够，且你愿意接受 **1.5-2 倍** 的 Token 成本。
- 错误类型主要是**格式、搜索策略、过度具体化**等反思能覆盖的问题。

如果任务是单次问答、成本敏感、或错误主要来自数字计算，Reflexion 的性价比不高。

### 2. 生产环境部署的简化建议

- **不要一上来就用 Reflexion**：先用 CoT 或 ReAct 建立基线，确认问题确实需要反思机制。
- **反思 Prompt 要具体**：让模型指出“具体哪个 Thought/Action 错了”，而不是泛泛而谈。
- **设置最大重试次数和 Token 上限**：避免陷入无限循环。例如限制 ReAct 8 步、Reflexion 只反思一次。
- **成本估算公式**：
  ```
  单次 Reflexion 预估 Token ≈ 1.5 × ReAct Token + 反思 Token
  ```
  按本实验数据，若 ReAct 单次约 4500 Token，Reflexion 约 7000 Token。乘以预计调用次数即可得到总预算。
- **考虑异步或缓存**：反思结果可以缓存到向量数据库，供相似问题复用，逐步降低长期成本。

## 结论

Reflexion 是一种优雅而实用的 Agent 增强机制。在本次 30 条 HotPotQA 样本的复现中，它把严格 EM 从 ReAct 的 36.7% 提升到了 60.0%，证明了“语言化反思”确实能帮助 Agent 从错误中恢复。

但这种提升是有代价的：接近 2 倍的 Token 消耗、对标准答案的依赖、以及在数字类问题上的局限。它更像是一位能即时复盘的学生，而不是一个真正学会了一类问题的专家。

对于工程师来说，Reflexion 的价值在于**低成本地提升交互式 Agent 的容错能力**。它不是万能药，但在合适的场景下，它能让你的 Agent 从“一错到底”变成“错一次，改一次”。
