## 任务目标

让平台自动生成一个 **LangGraph 多智能体文章写作系统**：

- 包含 researcher、writer、reviewer 三个角色；
- reviewer 从 accuracy、completeness、readability、structure 四个维度评分；
- 最多允许 3 次修改；
- 代码可运行，pytest 测试通过。

基线版本耗时 **5.3 小时**，得分 **16.5**，多次 600s 超时。最终版本耗时约 **12 分钟**，无超时，得分约 **15-23**。时间大幅改善，但功能正确性遇到瓶颈。

## 版本指标演进

| 版本 | 耗时 | 得分 | 超时 | 关键进展 |
|------|------|------|------|----------|
| v10 | 5.3h | 16.5 | 多次 | 单响应输出 6000+ token，频繁超时 |
| v11 | 14.3min | 23.0 | 0 | 文件数降到 8 个，chunk 拆分生效 |
| v12 | 11.7min | 18.0 | 0 | AST 校验通过，语义仍偏离 |
| v14 | 146s | 0 | 0 | `__init__.py` 导入不存在符号，快速失败 |
| v15 | ~30min | ~15 | 0 | agents.py 单文件 1699 行，graph.py 缺 ArticleGraph |
| v16 | ~12min | ~15 | 0 | pytest 可运行，1/4 通过 |

## 有效的策略

### 1. 文件级拆分

把一次 LLM 响应从"多个文件"改为"一个文件"，单次输出控制在 2000-4000 token，超时从 5.3 小时消失到 15 分钟以内。

### 2. 限制 file_plan 规模

禁止 pyproject.toml、.env.example、conftest.py、src/__init__.py、src/config.py，文件总数限制在 10 个以内，显著减少上下文噪音。

### 3. import 干跑

在 Docker 测试之前先做 import 干跑，零成本发现跨文件引用错误。

### 4. __init__.py 规则化生成

用 AST 扫描 `src/*.py` 中实际定义的顶层 Class/Function，再自动生成 import 语句，彻底解决`导入不存在符号`的问题。

### 5. 单文件行数/类方法数上限

强制每个文件不超过 200 行、每个类不超过 3 个方法，把三个 Agent 拆到单独文件，代码首次变得可导入、可运行。

## 无效或边际效应递减的策略

### chunk 拆分 + 续写合并

理论上能降低单次输出长度，但 LLM 容易在 chunk 中生成空方法体，拼接时又会出现缩进/try-except 断裂。不如直接把文件本身拆小。

### 硬编码字段名/类名

在 prompt 中强制"评分字段必须是 accuracy/completeness/readability/structure"对字段名有效，但无法保证状态流转、循环次数、异常抛出条件正确。

### 空方法体检测

最初用"方法体少于 5 行"作为 stub 标准，导致大量合法短方法被误判。简化为只检测 pass/空体/return None 后重试减少，但功能正确性问题仍未解决。

## 补充测试结果

为验证稳定性，做了三轮回归和一次压力测试：

| 测试 | 次数 | 平均耗时 | 平均得分 | 方差 | 超时 |
|------|------|----------|----------|------|------|
| Simple CLI | 3 | 4.91min | 82.0 | 0.0 | 0 |
| LangGraph | 3 | 19.26min | 12.33 | 54.06 | 1 |
| 大文件压力 | 1 | 22.6min | 0 | - | 0 |

Simple CLI 已非常稳定；LangGraph 方差大，说明当前模型对复杂状态机不可靠；大文件压力直接失败，验证了"单文件不能超过 300 行"的硬约束。

## 当前模型的能力边界

1. **单次输出上限**：稳定生成 3000-4000 token 的文件可行，超过 5000 token 截断概率显著上升。
2. **跨文件一致性**：无法可靠维护多文件接口约定，需要规则引擎兜底。
3. **状态机/循环逻辑**：对"最多修改 3 次"、"满足阈值退出"、"抛出 RevisionLimitExceeded"这类精确控制容易写错。
4. **测试驱动**：只能生成"通过自身测试"的代码，与隐藏测试存在 gap。
5. **长上下文利用**：prompt 中文件摘要过多时，会丢失细节，表现为字段名错误、方法体变短。

## 给下一次更换模型的建议

保留已被验证的架构设定：

- 每 group 1 个文件，单次输出 ≤ 4000 token；
- file_plan 文件总数 ≤ 10，禁止无关配置文件；
- 每个文件 ≤ 200 行，每个类 ≤ 3 个方法；
- import 干跑、`__init__.py` 规则化生成。

重点攻坚方向：

- 把 revision 次数限制的伪代码写进 prompt；
- 用 RAG 引入 LangGraph 多智能体示例；
- 尝试方法级生成 + AST 校验；
- 让 C 阶段拿到 evaluator 具体失败项，做针对性改写。

## 结语

这次调优让我意识到：**代码生成平台的瓶颈往往不在架构，而在模型对"精确逻辑"的遵循能力**。把架构、超时、语法错误都修好之后，剩下的就是和 LLM 的"听话程度"较劲。这份报告和补充测试数据，将作为下一次换模型时的基准起点。
