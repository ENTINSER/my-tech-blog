## 摘要

本文记录了一次 LangGraph 多智能体代码生成任务的调优过程。任务目标是让平台自动生成一个包含 researcher、writer、reviewer 三个角色、支持最多 3 次修改、且代码可运行并通过 pytest 的文章写作系统。基线版本耗时 **5.3 小时**，得分 **16.5**，且多次出现 600 秒超时；最终版本耗时稳定在约 **12 分钟**，无超时，得分约 **15–23**。

调优过程中，**文件级拆分、限制 file_plan 规模、import 干跑、`__init__.py` 规则化生成、单文件行数与类方法数上限**等策略被证明有效；而 chunk 续写合并、硬编码字段名、空方法体检测等策略效果有限或边际效应递减。本文基于版本演进数据与补充测试结果，讨论当前模型在代码生成任务中的能力边界，并为后续模型升级提供可保留的架构设定与重点攻坚方向。

## 1. 任务目标

本次调优的任务目标是让平台自动生成一个基于 LangGraph 的多智能体文章写作系统，具体要求如下：

- 包含三个 Agent 角色：researcher、writer、reviewer；
- reviewer 从 accuracy、completeness、readability、structure 四个维度对文章评分；
- 最多允许 3 次修改；
- 生成的代码可运行，并通过 pytest 测试。

评估指标包括任务耗时与 evaluator 综合得分。实验记录中的得分由 evaluator 基于上述四个维度计算得出，主要用于同一任务不同版本之间的相对比较。

## 2. 版本指标演进

下表展示了从基线到最终版本的关键指标变化：

| 版本 | 耗时 | 得分 | 超时 | 关键进展 |
|---|---:|---:|---:|---|
| v10 | 5.3h | 16.5 | 多次 | 单响应输出 6000+ token，频繁超时 |
| v11 | 14.3min | 23.0 | 0 | 文件数降到 8 个，chunk 拆分生效 |
| v12 | 11.7min | 18.0 | 0 | AST 校验通过，语义仍偏离 |
| v14 | 146s | 0 | 0 | `__init__.py` 导入不存在符号，快速失败 |
| v15 | ~30min | ~15 | 0 | agents.py 单文件 1699 行，graph.py 缺 ArticleGraph |
| v16 | ~12min | ~15 | 0 | pytest 可运行，1/4 通过 |

从 v10 到 v16，最显著的改进是**超时消失**与**耗时下降两个数量级**（从 5.3 小时到约 12 分钟）。得分方面，v11 达到峰值 23.0，但后续版本在功能正确性上遇到瓶颈，得分在 15–23 之间波动。

## 3. 有效的调优策略

### 3.1 文件级拆分

将一次 LLM 响应从“输出多个文件”改为“输出一个文件”，单次输出控制在 2000–4000 token。该策略直接消除了因单响应过长导致的 600 秒超时，使耗时从 5.3 小时降至 15 分钟以内。

### 3.2 限制 file_plan 规模

禁止生成 `pyproject.toml`、`.env.example`、`conftest.py`、`src/__init__.py`、`src/config.py` 等辅助文件，并将文件总数限制在 10 个以内。该约束显著减少了上下文噪音与不必要的生成开销。

### 3.3 import 干跑

在 Docker 测试之前先执行 import 干跑，以零 Docker 启动成本发现跨文件引用错误。该策略在 v14 阶段暴露出 `__init__.py` 导入不存在符号的问题。

### 3.4 `__init__.py` 规则化生成

用 AST 扫描 `src/*.py` 中实际定义的顶层 Class 与 Function，再自动生成 import 语句。该策略彻底解决了“导入不存在符号”导致的快速失败问题。

### 3.5 单文件行数与类方法数上限

强制每个文件不超过 200 行、每个类不超过 3 个方法，并将三个 Agent 拆分到单独文件。该约束使代码首次变得可导入、可运行，为后续 pytest 测试通过奠定了基础。

## 4. 无效或边际效应递减的策略

### 4.1 chunk 拆分 + 续写合并

理论上，chunk 拆分可以降低单次输出长度。但实际应用中发现：LLM 容易在 chunk 中生成空方法体，拼接时又会出现缩进或 `try-except` 结构断裂。最终效果不如直接将文件本身拆小。

### 4.2 硬编码字段名/类名

在 prompt 中强制要求评分字段必须是 `accuracy/completeness/readability/structure`，对字段名规范有效，但无法保证状态流转、循环次数、异常抛出条件等动态逻辑的正确性。

### 4.3 空方法体检测

最初以“方法体少于 5 行”作为 stub 判断标准，导致大量合法短方法被误判。简化为只检测 `pass`、空体、`return None` 后重试次数减少，但功能正确性问题仍未解决。

## 5. 补充测试结果

为验证稳定性，进行了三轮回归测试与一次大文件压力测试：

| 测试 | 次数 | 平均耗时 | 平均得分 | 方差 | 超时 |
|---|---:|---:|---:|---:|---:|
| Simple CLI | 3 | 4.91min | 82.0 | 0.0 | 0 |
| LangGraph | 3 | 19.26min | 12.33 | 54.06 | 1 |
| 大文件压力 | 1 | 22.6min | 0 | - | 0 |

### 5.1 观察

- **Simple CLI** 已非常稳定，平均得分 82.0，无超时，方差为 0。
- **LangGraph** 平均得分 12.33，方差 54.06，且出现 1 次超时，说明当前模型对复杂状态机的生成能力不稳定。
- **大文件压力测试** 直接失败，得分 0，验证了“单文件不能超过 300 行”的硬约束是必要的。

## 6. 当前模型的能力边界

基于上述实验，当前模型在以下方面表现出明显边界：

1. **单次输出上限**：稳定生成 3000–4000 token 的文件可行，超过 5000 token 时截断概率显著上升。
2. **跨文件一致性**：无法可靠维护多文件接口约定，需要规则引擎（如 AST 校验、自动 `__init__.py`）兜底。
3. **状态机/循环逻辑**：对“最多修改 3 次”、“满足阈值退出”、“抛出 RevisionLimitExceeded”等精确控制容易写错。
4. **测试驱动能力**：只能生成“通过自身测试”的代码，与隐藏测试之间存在 gap。
5. **长上下文利用**：prompt 中文件摘要过多时，会丢失细节，表现为字段名错误、方法体变短。

## 7. 给下一次模型升级的建议

### 7.1 应保留的架构设定

以下策略已被验证有效，应在更换模型时保留：

- 每 group 1 个文件，单次输出 ≤ 4000 token；
- `file_plan` 文件总数 ≤ 10，禁止无关配置文件；
- 每个文件 ≤ 200 行，每个类 ≤ 3 个方法；
- import 干跑与 `__init__.py` 规则化生成。

### 7.2 重点攻坚方向

- 将 revision 次数限制的伪代码明确写入 prompt；
- 用 RAG 引入 LangGraph 多智能体示例；
- 尝试方法级生成 + AST 校验；
- 让 Critic 阶段拿到 evaluator 的具体失败项，进行针对性改写。

## 8. 结论

本次调优将 LangGraph 多智能体代码生成任务从 5.3 小时、多次超时的不可用状态，优化到约 12 分钟、可运行 pytest 的稳定状态。这一改进主要归功于对单次输出长度、文件规模与跨文件一致性的严格约束，而非模型能力的根本提升。

实验结果表明，当前代码生成平台的瓶颈往往不在架构设计，而在模型对精确逻辑与跨文件接口约定的遵循能力。在架构、超时、语法错误等工程问题被修好之后，剩余的核心挑战是如何让 LLM 更稳定地生成功能正确的状态机与循环控制代码。

本次调优过程中形成的约束规则、失败案例与补充测试数据，可作为下一次模型升级时的基准起点。
